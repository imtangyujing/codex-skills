# 状态契约

本文件定义AIGC峰会SOP各模块共享的状态表。任何模块完成自己的最后一步后都必须写入状态；失败时必须写入`失败`，并补齐`error`和`retry_reason`。

## 状态表形态

Feishu Base状态表是流程跟踪工作面。每个模块完成后，都要把当前状态写入Base。

每一行代表一个可跟踪对象，包括总控、单人稿、圆桌稿和圆桌嘉宾。用户查看进度时只看Feishu Base。

## 状态流

1. `待建稿`
2. `已建总稿骨架`
3. `待骨架确认`
4. `已骨架确认`
5. `已建单人空文档`
6. `待妙记`
7. `已发现妙记`
8. `已拉取逐字稿`
9. `已写入总稿`
10. `待人工确认`
11. `已人工确认`
12. `已同步单人文档`
13. `已导出Word`
14. `已交付`

失败状态统一写为`失败`。失败事件必须包含：

```json
{
  "status": "失败",
  "error": "具体错误",
  "retry_reason": "修复或重试条件"
}
```

## Feishu Base状态表

创建Feishu Base，并在每个模块结束时同步状态表。

推荐表名：`状态跟踪`。

推荐字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| 对象 | text | 总控、单人嘉宾或圆桌嘉宾名称 |
| 类型 | select | `总控`、`单人稿`、`圆桌稿` |
| 状态 | select | 使用本文件状态流 |
| 负责人 | text | 分工负责人 |
| 角色 | text | 单人演讲、GenTalk、圆桌对话等 |
| 主题 | text | 分享主题 |
| 文档链接 | text/url | 总稿、单人稿或圆桌稿链接 |
| 总稿链接 | text/url | 总稿链接 |
| 更新时间 | datetime | 状态更新时间 |
| 备注 | text | 错误、重试提示或阶段说明 |

同步命令：

```bash
python3 scripts/base_status_sync.py \
  --state <state_source> \
  --assignment-input assignment_input.json \
  --base-token <base_token> \
  --table-id <table_id> \
  --total-doc-url <total_doc_url>
```

首次同步前使用`--dry-run`检查行数、状态、链接和圆桌归属。

## 建稿模块约定

建稿入口模块先写总稿骨架，并写入`待骨架确认`。在用户明确确认前，不创建单人空文档或圆桌文档。

总控行成功后写`已建总稿骨架`，等待确认时写`待骨架确认`。收到用户确认并进入`已骨架确认`后，为单人稿和圆桌稿写入对应跟踪行，成功后进入`待妙记`。
